# Migrating a pre-samohost inline-Caddyfile VM (e.g. field-record)

Some VMs (notably field-record) were bootstrapped before samohost's sites.d
pattern and have inline site blocks directly in `/etc/caddy/Caddyfile` rather
than using `import sites.d/*.caddy`. The heal script detects this at runtime
and exits with `inline-caddyfile` (never writes anything) until the migration
is done.

## Prerequisites

- SSH access to the VM (e.g. `ssh -p 2223 agent@178.105.246.151` for field-record)
- The VM must have the four Caddy sudoers grants (see heal-enable-runbook.md §2)
- A timestamped backup command available: `cp -a /etc/caddy /root/caddy-backup-$(date +%s)`

## Steps (run on the VM as root or via sudo)

### 1. Take a backup

```sh
sudo cp -a /etc/caddy /root/caddy-backup-$(date +%s)
```

### 2. Generate the sites.d vhost content

On the control plane, dry-run the heal to get the expected vhost content:

```sh
samohost app heal <vm> <app>   # no --apply
```

Copy the generated `00-main-<app>.caddy` content from the script output.

### 3. Install the sites.d vhost

```sh
sudo mkdir -p /etc/caddy/sites.d
# Write the generated content (include the provenance header as line 1):
sudo tee /etc/caddy/sites.d/00-main-<app>.caddy <<'EOF'
# generated by samohost/src/caddy/render.ts — do not edit by hand
<address> {
  ... (content from dry-run output)
}
EOF
```

### 4. Validate before removing the inline block

```sh
caddy validate --config /etc/caddy/Caddyfile
```

This will **fail** with a duplicate-site error while the inline block still
exists. That is expected. Proceed to step 5.

### 5. Remove ONLY the app's inline site block from Caddyfile

Edit `/etc/caddy/Caddyfile` and remove the inline `<address> { ... }` block
for the app being migrated. Keep all other inline blocks (e.g. the
`console.field-record-1.samo.team` block which has no sites.d equivalent).

If the `import sites.d/*.caddy` line is absent, add it:

```
import sites.d/*.caddy
```

### 6. Validate again (must succeed now)

```sh
caddy validate --config /etc/caddy/Caddyfile
```

### 7. Reload and probe

```sh
sudo systemctl reload caddy
curl -sk -H "Host: <app-mainHost>" https://127.0.0.1/ -o /dev/null -w "%{http_code}\n"
```

Expected: `200`.

### 8. Adopt provenance (if vhost was hand-written pre-heal)

If the sites.d file was written by hand (no provenance header on line 1), run:

```sh
samohost app heal <vm> <app> --apply --adopt-provenance --json
```

Expected outcome: `adopted`. Re-run without `--adopt-provenance` → `no-drift`.

### 9. Final idempotence proof

```sh
samohost app heal <vm> <app> --apply --json
# Expect: no-drift
samohost app heal <vm> <app> --apply --json
# Expect: no-drift (second run)
```

### Rollback

If anything goes wrong, restore from the backup:

```sh
sudo cp -a /root/caddy-backup-<timestamp>/. /etc/caddy/
caddy validate --config /etc/caddy/Caddyfile
sudo systemctl reload caddy
```
